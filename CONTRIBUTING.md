# Contributing to Diffract

Thank you for your interest in the project. Issues and pull requests are
welcome.

## Reporting Issues

Open a GitHub issue with your Python version, the `diffract-core` version,
and a minimal snippet that reproduces the problem. For security issues,
follow [SECURITY.md](SECURITY.md) instead of opening a public issue.

## Development Setup

Diffract develops against Python 3.12 and 3.13 with
[uv](https://docs.astral.sh/uv/):

```bash
git clone https://github.com/Risk-AI-Research/diffract.git
cd diffract
uv sync --extra dev --extra torch
```

## Design Principles

These choices are intentional; contributions should preserve them.

- **Kernels read like formulas, and the signature is the contract.** A
  kernel body is the numerical formula and nothing else — a mathematician
  should be able to open it and recognize the math. Parameters without
  defaults are input fields; keyword parameters with defaults are
  user-tunable config; `*args`/`**kwargs` are rejected. Orchestration
  (chunking, prefetching, parallelism, result writing) lives in
  `execution/` and is selected declaratively via `apply_level` and
  `execution_protocol`, never inside a kernel body.
- **The dataflow graph comes from field names.** Kernels declare what they
  require and produce by field name; the registry resolves the dependency
  chain, and every field has exactly one producer. Variants of the same
  math are additional `@kernel` registrations stacked on one function, not
  copies of its body.
- **Heavy numerics live in `extensions/`, with scholarly docstrings.**
  Literature references, validity domains, known biases of approximations,
  and determinism guarantees belong in the extension module docstring;
  kernels stay thin adapters over it.
- **Only the current chunk lives in RAM.** Proxies and views are handles,
  not containers; work is split by the cache's byte budget and results
  stream to storage as they are produced. Correctness must never depend on
  a cache hit, and framework tensors never survive extraction.
- **The bare core always works.** `import diffract` must not initialize —
  or be able to fail on — any optional runtime. Kernels that need an extra
  register only when it is usable, automatic paths degrade gracefully,
  explicitly requested ones fail loudly, and every missing-dependency
  error names the exact extra to install.
- **The lifecycle is explicit and errors are actionable.** Adding,
  computing, exporting, and erasing are separate user-invoked steps;
  nothing computes or deletes implicitly, and destructive operations
  require unambiguous intent. Not-found errors suggest close matches and
  point to the introspection API.
- **Deterministic by default, verified against references.** Stochastic
  code takes an explicit seed with a fixed default and produces identical
  results across machines. Numerical claims are pinned by parity tests
  against named references, with golden files regenerated by committed
  scripts that record the reference versions.

## Checks

CI runs linting, formatting, the test suite on both supported Python
versions, and the docs build. Locally:

```bash
make lint        # ruff check
make format      # ruff format
make test-light  # unit tests and README examples
make test        # full suite; some integration tests skip unless the
                 # backing services (Redis, S3) are configured
make docs        # sphinx build
```

The coverage gate (`fail_under` in `pyproject.toml`) is a ratchet: it only
ever moves up. Raise it when a change lifts measured coverage; never lower it
to make a change pass — add the missing tests instead.

## Conventions

- Python code blocks in the README run as tests in CI — keep them
  executable.
- Every registered kernel is executed end to end by the smoke test, and the
  kernel manifest must match the snapshot in
  `tests/unit/core/compute/kernels/test_kernel_smoke.py`. Adding a kernel
  means updating the snapshot in the same change.
- The coverage gate is enforced by `make test`; new code should come with
  tests.
- Commit messages follow [Conventional Commits](https://www.conventionalcommits.org/)
  (`feat:`, `fix:`, `docs:`, ...).

## Pull Requests

Keep pull requests focused on one change, include tests for new behavior,
and make sure CI is green. For larger changes, opening an issue first to
discuss the direction saves everyone time.
